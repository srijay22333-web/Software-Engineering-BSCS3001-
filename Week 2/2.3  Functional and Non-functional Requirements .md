# Week 2 — Lesson 3: Functional and Non-Functional Requirements

## Lesson Overview

Welcome back! By now you know how to identify users and gather requirements from them using interviews, questionnaires, focus groups, observation, and documentation. Here is the simple version of what comes next: once you have a big pile of requirements, you will notice they are not all the same "type" of requirement. Think of it like ordering a meal at a restaurant. One part of your order is WHAT you want — "bring me a plate of pasta." Another part of your order is HOW WELL you want it done — "make sure it arrives within fifteen minutes, while it is still hot." Both are real requirements, but they are completely different in nature. This lesson teaches you to tell these two types apart, called functional requirements and non-functional requirements, using the same Amazon Seller Portal example you have already seen, plus a hospital booking system and an online banking system. This is actually one of the most important distinctions you will learn in this entire course, so let's break it down carefully.

---

## Concept 1: Functional Requirements

### Simple Definition
A **functional requirement** describes what the system should do — it captures a specific action a user performs, and the specific output the system should produce in response. Think of it like a vending machine: you press a button (input), and a specific snack drops down (output).

### Why It Exists
Functional requirements exist because every software system needs a clear list of the actual features and actions it must support. Without writing these down clearly, developers would not know which buttons, screens, or actions they actually need to build. Functional requirements give the development team a precise, checkable list of "this input should produce this output," so there is no confusion about what the software is supposed to actually do.

### Why It Matters to You
If functional requirements are not clearly written, developers might build the wrong actions entirely, or miss key actions a user needs to perform. For example, if nobody specifies that a seller needs to be able to delete a product from their catalog, and not just add one, sellers will be stuck with an Amazon Seller Portal that has no way to remove outdated or sold-out products, which is a serious usability gap.

### Intuition — Understand It Without Jargon
Think about an ATM machine. You insert your card, type in an amount, and press "withdraw" — this is the input. The ATM then gives you exactly that amount of cash — this is the output. You do not care, at this exact moment, about how fast the ATM processes your request or how reliably it works every single day; you only care that pressing "withdraw 500" gives you exactly 500 in cash. This input-leads-to-output relationship is exactly what a functional requirement captures: a clear action by the user, and a clear resulting behavior from the system.

### How It Works — Step by Step
1. Identify a specific action that a user needs to perform with the system. For example, in the Amazon Seller Portal, a seller needs to manage their catalog.
2. Break this action down into its precise input and output. For example, the seller selects "add item" and provides product details (input), and the system adds that product into the catalog (output).
3. Write the functional requirement so it clearly describes both the action and the result, such as "a seller can add or delete items from their catalog."
4. Repeat this process for every distinct action the system needs to support, since each separate action is usually its own functional requirement.
5. Remember the mathematical analogy from the lesson: a functional requirement works like a mathematical function, transforming an element from the input domain into a value in the output, just like a function takes an input and produces an output.

### Real-World Example
Think about the **Amazon Seller Portal**. A seller wants to update what they are selling, so the system must let them add a new item, edit the price or description of an existing item, or delete an item that is no longer for sale. Each of these is a clear action with a clear, expected result — add an item and it appears in the catalog, delete an item and it disappears from the catalog. This is an example of a functional requirement because it describes a specific input (the seller's action) and a specific output (the change in the catalog), exactly what the system should do.

### Common Mistake
You might think that any requirement mentioning a feature of the system is automatically a functional requirement. This is not always true — only requirements with a clear input-leads-to-output action are functional. If a requirement instead talks about speed, reliability, or how well something performs, it is not functional, even if it is connected to the same feature. For example, "a seller can add a product" is functional, but "the added product must appear within five seconds" is not, even though both are about the same catalog feature.

### Exam Tip
The exam often gives you a list of requirements and asks you to label each one as functional or non-functional. Use the official exam classification trick: if a requirement uses action verbs like "view," "add," "delete," "notify," "search," or "track," it is very likely functional, since these verbs describe a clear action and result.

### Remember This
If you can describe it as "user does X, system gives Y," it is a functional requirement.

## Concept 2: Non-Functional Requirements

### Simple Definition
A **non-functional requirement** specifies how well the system should behave, rather than what specific action it should perform. It does not describe a clear input-to-output action; instead, it describes a quality the system's behavior must have, like speed or reliability.

### Why It Exists
Non-functional requirements exist because simply having a feature work is not enough — the feature also needs to work well. Two systems can have the exact same functional requirement, like "add a product to the catalog," but one system might add it instantly while another takes thirty seconds. Non-functional requirements were created to capture this difference in quality, which functional requirements alone cannot describe.

### Why It Matters to You
If you ignore non-functional requirements, you might technically deliver every feature the customer asked for, and yet the customer could still be completely dissatisfied because the system is slow, unreliable, or crashes under pressure. This matters because, in many real products, functional requirements are simply necessary just to have a working app at all, while non-functional requirements often become the actual reason users prefer one app over a competing app that offers the exact same features.

### Intuition — Understand It Without Jargon
Think about a car. Every car's basic job, or function, is to drive you from point A to point B — that is the functional part, and almost every working car can do this. But what actually makes one car feel better than another is how smoothly it drives, how quickly it can stop, and how rarely it breaks down — these are non-functional qualities. Two cars can both "drive you from A to B," which is the same function, but a smoother, safer, more reliable car is clearly the better car because of its non-functional qualities, not because it does something fundamentally different.

### How It Works — Step by Step
1. Take a functional requirement that has already been identified, such as "a seller can add a product to the catalog."
2. Ask whether there is an expected quality of how well this action should happen, beyond just whether it happens at all.
3. Specify this quality with a clear, measurable standard. For example, "when a new product is added to the catalog, it must show up on the user's interface within five seconds."
4. Notice that this new requirement cannot be expressed as a simple input-to-output function, since it is describing behavior over time and under conditions, not a single transformation.
5. Categorize this requirement as non-functional, since it specifies how the system should behave rather than what action it performs.
6. Repeat this process across the system to capture other important qualities, such as reliability and robustness, which you will learn about next.

### Real-World Example
Think about the **Amazon Seller Portal** again. The functional requirement says a seller can add a new product to their catalog. The non-functional requirement says that once added, this new product must actually appear on the seller's screen within five seconds, not thirty seconds or a full minute later. The seller does not just want the feature to exist; they want it to feel fast and responsive while they are actively managing their store. This is an example of a non-functional requirement because it specifies a quality standard, "within five seconds," about how the system behaves, rather than describing a brand-new action the system performs.

### Common Mistake
You might think the word "non-functional" means the system is "not working" or "broken," since that is how we normally use the word "non-functional" in everyday English. This is exactly the kind of confusion the lesson warns you about. A non-functional requirement does not mean something is broken — it simply means the requirement describes a quality of behavior, like speed or reliability, rather than a specific action and result.

### Exam Tip
Use the official exam classification trick from the lesson's slides: ask yourself, does this requirement describe WHAT the system does (a feature or action)? If yes, it is functional. Does it describe HOW WELL the system does it (a quality or performance standard)? If yes, it is non-functional. As a quick shortcut, numbers like "500 users," "3 seconds," or "99.9% uptime" usually signal a non-functional requirement.

### Remember This
Functional requirements are necessary just to have a working app — non-functional requirements are often the differentiator that decides which app users actually prefer.

## Concept 3: Reliability

### Simple Definition
**Reliability** is the extent to which a program behaves the same way over time, in the same operating environment. In simple words, a reliable system gives you the same correct behavior every single time you use it under the same conditions.

### Why It Exists
Reliability exists as a non-functional requirement category because software that works correctly once is not good enough — it needs to keep working correctly every time it is used, for as long as it is in operation. Early software systems that crashed often or behaved inconsistently caused major frustration and loss of trust, so reliability was identified as a critical quality that must be explicitly planned for, not just hoped for.

### Why It Matters to You
If a system is not reliable, users will get frustrated very quickly, since they cannot trust the software to behave the same way each time. This matters even more for a system like the Amazon Seller Portal, where the inventory is constantly changing as customers buy and sell products — the portal must handle all of these operations consistently and correctly, without crashing, no matter how often the data changes.

### Intuition — Understand It Without Jargon
Think about a simple alarm clock. A reliable alarm clock rings at exactly 7:00 AM every single morning, without fail, as long as you set it correctly and the conditions (like having battery power) stay the same. If the alarm clock sometimes rings at 7:00 AM, sometimes at 7:15 AM, and sometimes does not ring at all, you would call it unreliable, even though it technically has the "ring at a set time" function. This maps directly onto software reliability: the software should behave the same correct way every time, under the same conditions, just like a dependable alarm clock.

### How It Works — Step by Step
1. Identify a core operation that the system performs repeatedly, such as updating inventory when an item is sold.
2. Define the expected, correct behavior for this operation under normal, repeated use.
3. Ensure the system produces this same correct behavior every time the operation happens, as long as the operating environment stays the same.
4. Test the system repeatedly under the same conditions to check that it does not crash or behave inconsistently.
5. Pay special attention to high-frequency operations, since the Amazon Seller Portal's inventory changes constantly due to ongoing buying and selling, making consistent behavior especially important here.

### Visual Diagram
```
+--------------------------------------------------------------+
|                    RELIABILITY OVER TIME                       |
+--------------------------------------------------------------+

   DAY 1                DAY 2                DAY 3
   +-----------+        +-----------+        +-----------+
   | Inventory   |       | Inventory   |       | Inventory   |
   | updates       |      | updates       |      | updates       |
   | correctly,    | ---> | correctly,    | ---> | correctly,    |

   SAME OPERATING ENVIRONMENT + SAME CORRECT BEHAVIOR = RELIABLE
```
This diagram shows that a reliable system performs the same correct behavior consistently, day after day, under the same operating environment in plain English.

### Real-World Example
Think about an **online banking** system. Customers expect that every single time they check their account balance, the number shown is correct and the app does not crash, whether they check it at 9 AM or 11 PM, on a weekday or weekend. Banks often promise their systems will be available and working correctly 99.9% of the time, which is a direct, measurable promise about reliability. This is an example of reliability because the bank is committing to the system behaving the same correct way, consistently, over time, in the same operating environment.

### Common Mistake
You might think reliability and robustness are basically the same thing, since both deal with the system "not breaking." This is incorrect. Reliability is about the system behaving consistently and correctly over time under normal, expected conditions. Robustness, which you will learn next, is specifically about how the system handles unexpected or erroneous situations, like bad input or unusually high traffic. They are related, but they are not the same quality.

### Exam Tip
Be ready to write the exact definition from the lesson, word for word if possible: reliability is the extent to which a program behaves the same way over time in the same operating environment. Exams often test this definition directly, along with the seller portal inventory example.

### Remember This
Reliability means doing the same correct thing, the same way, every single time, like a trustworthy alarm clock.

## Concept 4: Robustness

### Simple Definition
**Robustness** is the extent to which a program can recover from errors and unexpected input, without crashing or breaking. A robust system can handle being given something it did not expect, and still keep working.

### Why It Exists
Robustness exists because, in the real world, users do not always behave exactly as developers expect. Sometimes users accidentally type the wrong kind of input, like letters instead of numbers, or a system might suddenly face much higher traffic than usual. Robustness was identified as a critical quality because software that simply crashes or breaks the moment something unexpected happens is unreliable and frustrating, no matter how well it works under perfect conditions.

### Why It Matters to You
If a system is not robust, even a small mistake by a single user, like entering garbage input by accident, could crash the entire system, ruining the experience for everyone using it. This matters even more for systems like the Amazon Seller Portal, which must handle large numbers of transactions and high traffic, especially during busy shopping periods, without breaking down under the pressure.

### Intuition — Understand It Without Jargon
Think about a sturdy phone case. If you accidentally drop your phone, a good, robust phone case absorbs the shock and protects the phone so it still works perfectly afterward. A cheap, flimsy case might let the phone crack and stop working entirely after the very same drop. The "drop" here is like an unexpected event or bad input that the system did not plan for, and robustness is exactly like that sturdy case — it lets the system "survive" the unexpected situation and keep functioning, instead of breaking down completely.

### How It Works — Step by Step
1. Identify the types of unexpected situations the system might face, such as a user entering garbage or invalid input, or the system facing unusually high traffic.
2. Plan for how the system should respond to each unexpected situation, instead of assuming it will never happen.
3. Build in error handling so that when something unexpected occurs, the system catches the problem gracefully instead of crashing.
4. Test the system specifically with bad input and high-traffic conditions, not just with normal, expected usage.
5. Confirm that the system can recover from these situations and continue operating, rather than failing completely.

### Visual Diagram
```
+--------------------------------------------------------------+
|                      ROBUSTNESS IN ACTION                      |
+--------------------------------------------------------------+

   UNEXPECTED INPUT                     WITHOUT ROBUSTNESS
   +----------------------+             +----------------------+
   |  User types letters    |           |  SYSTEM CRASHES        |
   |  into a "price" field   |  ----->   |  completely, no one    |
   |  that expects numbers    |          |  can use it             |
   +----------------------+             +----------------------+

   UNEXPECTED INPUT                     WITH ROBUSTNESS
   +----------------------+             +----------------------+
   |  User types letters    |           |  System shows a clear  |
   |  into a "price" field   |  ----->   |  error message and      |
   |  that expects numbers    |          |  keeps working           |
   +----------------------+             +----------------------+
```
This diagram shows the same unexpected bad input handled two different ways — crashing without robustness, versus recovering gracefully with robustness — in plain English.

### Real-World Example
Think about an **online banking** system. Suppose a customer accidentally types letters into a field meant for a money transfer amount, like typing "abc" instead of "500." A robust banking system will immediately catch this mistake and show a clear error message asking the customer to enter a valid number, instead of crashing the entire app. This is an example of robustness because the system is recovering gracefully from an unexpected, erroneous input, exactly as the lesson describes, rather than breaking down completely.

### Common Mistake
You might think robustness is only about handling huge amounts of traffic, since that is one of the lesson's examples. This is incomplete, because robustness covers both unexpected erroneous input from a single user, like garbage text in a form, and the system's ability to keep handling large numbers and high traffic without failing. Both situations test the same underlying quality: can the system recover instead of breaking.

### Exam Tip
Be ready to write the exact definition: robustness is the extent to which a program can recover from errors and unexpected input. Exams may give you a scenario, like "a user enters invalid data" or "the system suddenly faces very high traffic," and ask you to identify it as a robustness example.

### Remember This
Robustness means surviving the unexpected, like a sturdy phone case that protects your phone after an accidental drop.

## Concept 5: Other Non-Functional Requirement Categories

### Simple Definition
Beyond reliability and robustness, there are several other categories of non-functional requirements that describe different qualities of how well a system behaves, including **performance**, **portability**, **security**, and **interoperability**.

### Why It Exists
These additional categories exist because reliability and robustness do not cover every important quality a system might need. A system could be perfectly reliable and robust, and still be too slow, unable to run on certain devices, vulnerable to hackers, or unable to work together with other systems. Each of these additional categories captures a different dimension of system quality that matters for different reasons.

### Why It Matters to You
If you only focus on reliability and robustness and ignore these other categories, you might still end up with a system that frustrates users in other ways, such as being painfully slow, only working on certain devices, or leaking sensitive customer data. Considering all relevant non-functional categories together gives you a much more complete picture of what "good quality" really means for your specific system.

### Intuition — Understand It Without Jargon
Think about buying a smartphone. You would not just ask "does it make calls?" (a functional question). You would also ask: how fast does it run apps (performance), can I use the same charger and accessories across different phone brands (portability), is my personal data protected (security), and can it connect properly with my other devices like a smartwatch or laptop (interoperability)? Each of these questions checks a different quality of the phone, beyond just whether the basic function works, which is exactly how these four non-functional categories work for software.

### How It Works — Step by Step
1. **Performance** — Check how fast the system responds, how much work it can handle at once (throughput), and how efficiently it uses resources like memory or processing power. For example, a hospital booking system might need to process payments within three seconds, or support 500 simultaneous logins during a busy period.
2. **Portability** — Check whether the system can work across different platforms and operating systems, such as running correctly on both Android and iOS devices, or on different computer operating systems.
3. **Security** — Check how well the system protects data, including authentication (confirming who a user is) and authorization (confirming what that user is allowed to do). For example, an online banking system must strictly protect customer account data from unauthorized access.
4. **Interoperability** — This term was mentioned in the lesson but not explained here. It will be covered later.
5. Combine all relevant categories — reliability, robustness, performance, portability, security, and interoperability — when specifying the full set of non-functional requirements for a real system.

### Real-World Example
Think about a **hospital booking system**. It must process patient payments within three seconds, which is a performance requirement, and it must be able to support 500 simultaneous logins during peak hours, such as during flu season, without slowing down. It must also protect sensitive patient medical records from unauthorized access, which is a security requirement. This is an example of multiple non-functional requirement categories working together because the hospital booking system needs to be fast, capable of handling many users at once, and secure, all at the same time, beyond just having the right features.

### Common Mistake
You might think performance and reliability mean the same thing, since both seem to be about the system "working well." They are different: reliability is about the system behaving the same correct way consistently over time, while performance is specifically about speed, throughput, and efficient use of resources. A system could be reliable (always gives correct results) but still have poor performance (it is just very slow every time).

### Exam Tip
You are less likely to be asked for deep detail on portability, security, or interoperability in this lesson, since the lecture only introduces them briefly. Focus your exam preparation on being able to name all of these categories and briefly state what each one checks, using the one-line definitions given here.

### Remember This
Performance is about speed, portability is about working everywhere, and security is about protecting data — three more lenses for checking system quality beyond reliability and robustness.

## Exam Alert — How to Quickly Classify Any Requirement

This is a high-value shortcut directly from the lesson slides, so memorize it carefully.

- Ask: does the requirement describe WHAT the system does, like a feature or action? If yes, it is a **functional requirement**.
- Ask: does the requirement describe HOW WELL the system does something, like a quality or performance standard? If yes, it is a **non-functional requirement**.
- As a quick shortcut, numbers such as "500 users," "3 seconds," or "99.9% uptime" usually signal a non-functional requirement, since they describe a measurable standard of quality.
- As another quick shortcut, action verbs such as "view," "add," "delete," "notify," "search," or "track" usually signal a functional requirement, since they describe a specific action and result.

Here is a worked example using this exact shortcut, straight from the lesson slides:
- R1 = "Support 500 logins" → Non-Functional Requirement (a number describing capacity, a quality standard)
- R2 = "View doctors" → Functional Requirement (an action verb describing a feature)
- R3 = "Notify user" → Functional Requirement (an action verb describing a feature)
- R4 = "Process within 3 seconds" → Non-Functional Requirement (a number describing a speed standard)

## Key Takeaways

- Functional requirements describe WHAT the system should do, capturing a specific user action (input) and the system's resulting behavior (output), similar to a mathematical function.
- Non-functional requirements describe HOW WELL the system should behave, specifying quality standards rather than specific actions, and cannot be expressed simply as input leading to output.
- "Non-functional" does not mean broken or not working — it simply means the requirement is about a quality of behavior, not a specific feature.
- Reliability is the extent to which a program behaves the same way over time in the same operating environment, which matters most for systems with frequently changing data, like inventory in the Amazon Seller Portal.
- Robustness is the extent to which a program can recover from errors and unexpected input, such as garbage data or sudden high traffic, without crashing.
- Other non-functional categories include performance (speed and resource use), portability (working across platforms), security (protecting data), and interoperability (mentioned but not explained in this lesson).
- A simple exam shortcut is to look for action verbs (functional) versus numbers and quality standards (non-functional) when classifying a requirement.
- Functional requirements are necessary just to have a working system, while non-functional requirements are often what differentiates a good system from a great one.

### Potential MCQs

**1. Which of the following best describes a functional requirement?**
A) The system must be available 99.9% of the time
B) The system must support 500 simultaneous logins
C) A seller can add or delete items from their catalog ✔
D) The system must recover from invalid input without crashing

A seller adding or deleting catalog items is correct because it describes a clear user action and a clear resulting output, exactly what the lesson defines as a functional requirement. Availability percentage and login capacity are both non-functional, since they describe quality standards using numbers rather than actions. Recovering from invalid input describes robustness, a non-functional category, not a specific user action and output.

**2. Why can "the product must appear within five seconds" not be expressed as a simple mathematical function?**
A) Because it does not involve any user action at all
B) Because it describes a behavior standard over time, not a single input-to-output transformation ✔
C) Because it is not related to the catalog feature
D) Because it only applies to sellers, not buyers

The correct answer is that this requirement describes a quality standard for how the system behaves, not a single, clean input-to-output transformation like a functional requirement. It is incorrect to say no user action is involved, since the action (adding a product) still triggers this requirement. It is incorrect to say it is unrelated to the catalog feature, since it directly builds on the catalog functional requirement. It is also incorrect to say it only applies to sellers, since the requirement is about system behavior in general, not about a specific user type.

## Quick Memory Tricks

- **Functional Requirements:** Think "press button, get snack" — like a vending machine, a clear action always leads to a clear, expected result.
- **Non-Functional Requirements:** Think "necessary vs. differentiator" — functional features are necessary just to have a working app, but non-functional quality is often what makes one app better than another with the exact same features.
- **Reliability:** Picture a trustworthy alarm clock that rings at exactly the same time, every single day, without fail.
- **Robustness:** Picture a sturdy phone case that protects your phone after an accidental drop, letting it keep working instead of breaking.
- **Performance, Portability, Security:** Remember "Fast, Flexible, and Fort Knox" — performance is about being fast, portability is about being flexible across platforms, and security is about protecting data like a fortress.

---

## One-Line Lesson Summary

Functional requirements describe WHAT a system should do as a clear action-and-result pair, while non-functional requirements like reliability, robustness, performance, portability, and security describe HOW WELL the system should behave, and both types must be considered together to build software that truly satisfies users.
